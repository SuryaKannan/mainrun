# Repo Orientation: mainrun

> Generated by /orient:orient. Re-run to update.

## One-line purpose
Maincode's ML-engineering assessment: take a deliberately under-optimized GPT-2-style training run on Hacker News titles and lower its validation loss within a fixed 7-epoch budget (baseline 1.754; this submission reaches ~1.2046).

## Primary language(s)
Python (the model + training pipeline). A little JavaScript (small `zx`/Node scripts for the checkpoint and submit steps).

## Pipeline / workflow stages
The training run is orchestrated by `mainrun/train.py:main()`, in this order:
1. Load hyperparameters (`config.py` тАФ defaults, or a YAML override for sweeps).
2. Load + shuffle Hacker News titles, split into train/val (`data.py:get_titles`).
3. Train a byte-level BPE tokenizer; encode all titles joined by `<eos>` into one token stream (`data.py:train_tokenizer`, `BPETokenizer`).
4. Build the GPT model (`model.py:GPT`).
5. Train: AdamW + linear-warmupтЖТcosine LR schedule, gradient clipping, periodic `evaluate()` on the val stream.
6. Log one JSON event per step to `mainrun/logs/mainrun.log` (`logging_utils.py:DualLogger`); `analysis/` parses these into figures.

## Key files
- `mainrun/train.py` тАФ entry point. Hyperparameter wiring, the training loop, and the **frozen** `evaluate()`. Read this first; it shows the whole flow.
- `mainrun/model.py` тАФ the GPT itself: RoPE attention, per-title attention mask, GPT-2 residual init scaling. The architecture lives here.
- `mainrun/data.py` тАФ dataset loading, BPE tokenizer, and the batching helpers (`get_batch`, `iter_full_split`).
- `mainrun/config.py` тАФ `Hyperparameters` dataclass + `load_hyperparameters` (the optional YAML override).
- `mainrun/logging_utils.py` тАФ `DualLogger` and the JSON log schema that the grader and `analysis/` depend on.
- `mainrun/utils.py` тАФ the devcontainer gate; imported purely for its `sys.exit` side effect.
- `analysis/plot_runs.py` тАФ parses run logs into the journey/curves figures (`analysis/figures/`).
- `analysis/sweep.py` тАФ runs `train.py --config тАж` once per variation in a YAML spec, then prints a leaderboard.
- `tests/test_model.py` тАФ guards the parameter-count invariant and the per-title mask behavior.
- `Taskfile.yml` тАФ the `task train` / `task test` / `task sweep` / `task submit` commands.

## Core concepts
**The frozen `evaluate()`** : the assessment rules forbid changing it. It sums per-token cross-entropy over the val stream and divides by the val text's *character* count (an unusual normalization). It lives inside `train.py:main()`.

**RoPE (rotary positional embeddings)** : position is encoded by *rotating* the query/key vectors by an angle proportional to their position, inside attention, rather than adding a learned position vector. See `model.py:rotate_half` / `apply_rotary` and the buffers built in `CausalSelfAttention.__init__`.

**Per-title (document) attention mask** : each token may attend only within its own title (segments split at `<eos>`) and only to the past. The mask is built from the input token ids inside `model.py:GPT.forward`, so `evaluate()` and the data path stay untouched.

**GPT-2 residual init scaling** : the projections that write back into the residual stream are initialized with std `0.02 / sqrt(2 * n_layer)` so the stream's variance stays roughly constant with depth. See the loop at the end of `model.py:GPT.__init__`.

**Grader-safe config** : `python train.py` with no args uses the canonical defaults (the submitted config); `--config` only overrides for sweeps. `config.py` + the `__main__` block in `train.py`.

## Common gotchas
- **`import utils` exits outside the devcontainer.** `train.py` imports `utils`, which calls `sys.exit(1)` if `/root/.mainrun` is absent. That's why the tests never import `train.py`/`utils` тАФ they import `model`/`data`/`config` directly after a `sys.path` bootstrap (`tests/test_*.py`).
- **Determinism is a trap.** Runs are bit-reproducible under the fixed seed (1337). So two "different" configs that produce the *identical* loss means the config never actually changed (a real failure we hit when an edit didn't reach the trainer). Always check the `hyperparameters_configured` log line.
- **`evaluate()` depends on `batch_size`** (via `iter_full_split`), so validation losses at different batch sizes aren't perfectly comparable тАФ a measurement artifact, not a model-quality difference.

## Suggested exercise sequence
1. Open `mainrun/train.py` and read `main()` top to bottom (you can skip the internals of `evaluate()`). Then, without looking, list the stages a run passes through from "load config" to "logged validation loss," in order. Check yourself against the **Pipeline / workflow stages** section above.
2. Open `mainrun/model.py` and find `GPT.forward`. Read the first ~8 lines (the mask construction, up to `attn_mask = тАж`). What does the `seg` tensor represent for each token position, and what does the resulting mask stop a token from doing? Then find where `attn_mask` is actually *used* (hint: it's passed down through the blocks).
